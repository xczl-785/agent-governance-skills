# Agent First Operating Rules

这份文档用于承载可持续扩充的 agent-first 运行规则。

它不是某一个具体 skill 的触发说明，而是跨 skill、跨项目都可以复用的协作原则。后续如果某一组规则形成稳定触发条件、执行循环和输出契约，再拆分为独立 skill。

## 避免最小化方案

若存在标准方案，应采用完整方案。禁止把临时方案或过渡方案当作最终交付。

要求：

- 以标准方案为目标设定计划。
- 可以分阶段推进，但每个阶段都要服务于最终标准方案。
- 不要求在计划前探清所有情况；在已知信息足够时，应选择更优、更合适的方案。
- 如果必须暂时采用过渡步骤，需要说明它通向哪个标准方案阶段，以及何时被替换或吸收。

## 问题持续存在

在规划阶段、任务拆分或阶段推进时，不只记录要做的任务，也要记录暂不处理的任务。

暂不处理的任务需要有明确归属：

- 记录位置。
- 计划在哪个阶段解决。
- 如果暂时没有归属，需要标注后续处理方式。

不要让已知问题因为本阶段不处理而变成隐形问题。

## 协作规则

推进阶段性工作时，应区分总控与长线执行角色。

推荐模型：

- 一个 agent 作为总控窗口，负责调度、整体规划、阶段边界和上下文归档。
- 总控另行调起规划、实施、审查三个长线 agent，分别负责具体、独立、分离的工作。
- 阶段完成、切换下一阶段，或某个 agent 上下文超载时，可以关闭该 agent。
- 阶段切换时通常重新开启规划、实施、审查 agent，再从阶段交接信息继续推进。

这个模型适合较大、较长、需要多角色校验的 agent-first 工作；不适合简单单步任务。

## 维护方式

新增规则时优先放在本文件，直到它满足拆分为 skill 的条件。

可以拆分为 skill 的信号：

- 有明确的触发场景。
- 有稳定的执行循环。
- 有可验证的输出契约。
- 能独立复用，不依赖某个私有项目语境。

如果只是价值观、协作偏好或通用约束，继续留在本文件。
